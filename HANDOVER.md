# Web3 Builder Hub - 项目交接与状态文档

## 🎯 当前项目情况总结

我们已经成功完成了项目第一阶段最核心的攻坚：**彻底打通并修复了“采集 (Collection) ➡️ 存储 (Storage) ➡️ 分析 (Analysis) ➡️ 展示 (Presentation)”的数据全流程闭环**。目前，系统已经具备了一个坚如磐石、完全透明的低成本 MVP 数据处理架构。

整个系统的设计理念是：**自动采集 → 入库排队 → 晚间定时批量大模型处理**。数据流转过程对用户 100% 可视、可控、可验证，彻底消除了数据流转的“黑盒”状态。

## ✅ 已完成工作

### 1. 数据管道验证面板 (/data)
- 构建了专用的数据全景验证控制台，分为 `COLLECTION`, `STORAGE`, `ANALYSIS`, `PRESENTATION` 四个核心模块。
- **透明度提升**：所有模块彻底实现数据驱动，告别硬编码。用户可以直接在前端查阅真实的大模型排队情况、解析日志、源站分布以及最终交付的评估分数。
- 修复了因为 React 直接渲染 JSON 对象导致的客户端报错。

### 2. 数据采集层 (Collection) 的健壮性修复
- **Tier 1 (RSS)**：彻底修复了部分博客（如 BNB Chain）由于输出不规范、未转义 `&` 符号导致底层 RSS 解析器崩溃的漏洞。
- **Alpha (API)**：果断放弃了在 Vercel 等 Serverless 云环境下极易触发 Cloudflare 盾或 WAF 拦截的中心化接口（如原有的 DoraHacks、Devfolio 前端接口）。全面迁移至对服务器 IP 极其友好的数据源：
  - **The Graph 节点**（如 Questbook 子图）。
  - **原生开放 REST API**（如 Taikai）。
  - **传统 SSR 页面无头爬取**（如 ETHGlobal）。
  - 为 GitHub Search API 补齐了 `GITHUB_PAT` 授权，避免了严苛的限流风控。

### 3. 核心流转机制与 LLM 漏斗修复
- **模型降级与兜底**：修复了在初筛阶段由于大模型 API 报错（如名称错误、接口限流）导致的 100% 静默数据丢弃（Silent Failure）的灾难性漏洞。现在的逻辑是：如果初筛 LLM 报错，将默认放行该批次数据，确保“宁可错杀，不能漏掉”。
- **修复 Fetch 缓存陷阱**：在 `triggerApi` Server Action 中引入了 `cache: 'no-store'` 和强制的 `POST` 方法，彻底根除了 Next.js 13/14 对 GET 请求的激进缓存，确保每次点击都是真实的物理采集和日志记录。
- **数据库 Schema 迁移补齐**：补齐了遗漏的 `drizzle` SQL 迁移脚本（为 `api_logs` 新增了 `skipped` 等字段），修复了线上日志写库崩溃却静默吞没报错的问题。
- **模型名称对齐**：根据要求，将项目中散落的模型配置统一还原为 Kimi 最原始的预览版代号（`kimi-k2.5`, `kimi-k2-turbo-preview`, `kimi-k2-0905-preview`）。

## ⏳ 未完成事项与已知问题

1. **部分数据源抓取质量需要微调**：
   目前数据流转管道虽然 100% 跑通，但在大模型极为精准的意图过滤下（如：必须是明确面向开发者的机会），CryptoFundraising 的早期融资新闻和部分 GitHub 搜索结果会被 LLM 过滤掉，这导致了部分抓取源入库数据偏少。
2. **定时任务 (Cron) 的全面挂载**：
   Vercel `vercel.json` 中的 Cron 任务调度逻辑已经写好（每天定时采集和 23:00 定时 Deep Dive 分析），但可能需要根据未来的数据量对时间频次和并发进行微调。
